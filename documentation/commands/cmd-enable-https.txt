~ Name:
~ ~~~~~
~ enable-https -- Enable HTTPS on port 9443 (HTTP/2 + HTTP/3 via ALPN)
~
~ Synopsis:
~ ~~~~~~~~~
~ play enable-https [app_path] [--regenerate]
~
~ Description:
~ ~~~~~~~~~~~~
~ Configures the application to listen for HTTPS on port 9443 alongside
~ the plain HTTP listener on port 9000. The framework activates ALL
~ three modern HTTP versions automatically when HTTPS is on:
~
~   • HTTP/1.1 over TLS (ALPN fallback for old clients)
~   • HTTP/2 over TLS (ALPN h2 — modern browsers, curl --http2, etc.)
~   • HTTP/3 over QUIC (UDP same-port; advertised via Alt-Svc header so
~     clients discover it after their first HTTPS request)
~
~ Graceful degradation: HTTP/3 needs the Netty native QUIC library,
~ shipped for osx-{aarch_64,x86_64}, linux-{aarch_64,x86_64}, and
~ windows-x86_64. On linux-riscv64 (or anywhere the native binary won't
~ link), the framework logs a WARN and skips the UDP listener; HTTPS,
~ HTTP/2, and HTTP/1.1 keep working normally on TCP.
~
~ Performs three steps idempotently:
~
~   1. If certs/host.cert and certs/host.key do not both exist, generate a
~      PEM cert+key pair. Two strategies:
~
~        • mkcert (preferred) — if `mkcert` is on PATH, runs
~          `mkcert -cert-file certs/host.cert -key-file certs/host.key
~                  localhost 127.0.0.1 ::1`
~          to produce a cert signed by mkcert's local CA. After running
~          `mkcert -install` once, the local CA is trusted by the
~          system store, which means Chrome's QUIC stack will accept it
~          and HTTP/3 actually negotiates in the browser. See
~          https://github.com/FiloSottile/mkcert.
~
~        • openssl fallback — if mkcert isn't found, runs
~          `openssl req -x509 -newkey rsa:2048 -nodes ...`
~          to produce a self-signed cert good for curl/h2 testing but
~          NOT trusted by browsers. The command prints a hint about
~          installing mkcert on this path.
~
~      Keys are unencrypted PEM (Netty's SslContextBuilder does not
~      accept encrypted keys). If the cert+key already exist, they are
~      reused.
~
~   2. Set certificate.file=certs/host.cert and
~      certificate.key.file=certs/host.key in conf/application.conf.
~      Skeleton-style commented lines are uncommented in place. ONLY
~      default values are written — existing active lines (e.g. a
~      custom certificate.file=conf/my-cert.pem or https.port=8443)
~      are preserved verbatim.
~
~   3. If http.port is not already an active line, set http.port=9000.
~      The framework's special-case "default to 9000 when neither port
~      is set" fallback (Server.java) does NOT fire once https.port is
~      configured, so the explicit http.port line is needed to keep the
~      plain HTTP listener bound. A user-customized http.port (e.g.
~      8080, or -1 to disable HTTP entirely) is preserved across
~      enable-https runs.
~
~ Run play run or play start afterwards to apply. The startup log will
~ show three listener lines:
~     Listening for HTTP on port 9000 (Waiting a first request ...)
~     Listening for HTTPS on port 9443 (Waiting a first request ...)
~     Listening for HTTP/3 on UDP port 9443 ...
~
~ Idempotent: re-running when HTTPS is already enabled is a no-op. The
~ command reuses an existing cert+key when both files are present and
~ the cert is still valid, so disable-https followed by enable-https
~ reuses the same cert.
~
~ Auto-regeneration: when openssl is on PATH, enable-https checks the
~ existing cert for expiry / corruption and regenerates it
~ transparently if it has expired or fails to parse. This is fine for
~ a localhost dev cert; manual recovery would just be busywork.
~
~ Forced regeneration: pass --regenerate to regenerate the cert+key
~ unconditionally, even if they're valid. Useful when you want a fresh
~ cert for any reason (e.g., changed your local hostname, want to
~ rotate keys, switched between mkcert and openssl).
~
~ Note: the bare --force flag is reserved by the play CLI itself
~ (consumed for the ignoreMissing path before commands see it), so
~ this command uses --regenerate instead.
~
~ The generated cert is for local development only — for production,
~ replace certs/host.cert and certs/host.key with a real cert (Let's
~ Encrypt, internal CA, etc.).
~
~ See also: play disable-https
~
